# AGENT 工作约定

本文记录 agent 在当前项目中协作时必须遵守的项目级约定。除非用户明确覆盖，后续 agent 应优先按本文件执行。

## 进入项目后的第一步

在当前目录开始任何检查、修改或提交准备前，先阅读：

- `TREE.md`
  - 作用：保存当前项目文件树，帮助快速定位文件和目录结构。
- `NAVIGATION.md`
  - 作用：保存当前项目完成情况、已实现功能，以及关键实现位置 address。

阅读后再决定具体改动范围，避免重复探索或误改历史兼容文件。

## CodeReview 和检查记录

当 agent 进行 CodeReview、bug 检查、回归检查、shellcode 验证或其他质量检查时：

1. 将检查结果写入 `REVIEW.md`。
2. 每条 finding 应包含问题位置、问题说明、修复动作和验证方式。
3. 如果 finding 已修复，在 `REVIEW.md` 中明确记录对应 fix。
4. 如果 finding 暂不修复，也要说明原因、风险和后续建议。
5. 不要用新的 review 文档替代 `REVIEW.md`；同一轮工作的审查记录应集中维护在 `REVIEW.md`。

## REVIEW.md 完成后的备份机制

当本轮 CodeReview、检查、修复和验证全部完成后，agent 必须触发 `REVIEW.md` 备份机制：

1. 确认 `REVIEW.md` 已经包含本轮全部检查结果、修复动作和验证结果。
2. 创建 `./backup/` 目录。
3. 统计 `./backup/` 中已有的 `Review-xx.md` 文件数量，并生成下一个备份文件名：
   - 文件名格式：`Review-xx.md`
   - `xx` 从 `01` 到 `99`
   - 编号使用两位数字，不足两位前面补 `0`
   - 默认取当前备份数量加一；如果目标文件已存在，使用下一个可用编号
4. 将当前 `REVIEW.md` 的完整内容保存到 `./backup/Review-xx.md`。
5. 确认备份文件写入成功后，清空原 `REVIEW.md`，保留空文件。
6. 如果编号已经超过 `99`，不要覆盖旧备份；应停止并向用户说明需要人工处理。

`backup/` 是本地归档目录，不进入版本控制，已在 `.gitignore` 中忽略。

## 修改代码时

- 保持改动聚焦，只修改完成当前任务所需的文件。
- 不要回滚用户或其他 agent 已经做过的无关改动。
- 对历史 decoder、`.bin`、`.asm`、Testival 文件保持谨慎，确认用途后再修改。
- 生成或运行测试产生的缓存文件不应纳入版本控制。

## 用户决定进行 git commit 时

当用户明确决定要进行 `git commit`，或要求准备提交内容时，agent 需要同步检查并按需更新：

- `NAVIGATION.md`
  - 更新项目完成情况、关键功能位置和验证记录。
- `TREE.md`
  - 更新当前项目文件树。
- `README.md`
  - README 大部分情况下不要大改，只同步必要的用户可见变化、用法变化或新增验证方式。

提交前还应确认 `REVIEW.md` 是否已经按本文件要求完成备份并清空。

## 推荐验证命令

根据改动范围选择运行：

```bash
python3 -m compileall -q .
python3 ALPHA3.py --help
python3 test/validate_x64_shellcode.py
```

如果只改文档，可不运行完整验证，但应在回复中说明未运行代码验证。
